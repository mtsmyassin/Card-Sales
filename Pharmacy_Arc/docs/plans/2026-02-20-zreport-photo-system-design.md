# Z-Report Photo System — Design Document
**Date:** 2026-02-20
**Status:** Approved
**Scope:** Enterprise photo system for Z-report reconciliation via Telegram bot

---

## Problem Statement

Photos sent via Telegram have no reliable way to be tied to a specific store/register/day. The current system stores a single `z_report_image_path` string inside the JSONB `payload` column of `audits`, which breaks for multi-store, multi-register operations and provides no photo metadata.

---

## Core Insight

**The photo is the primary record.** The audit entry is derived from it.
Manager sends a photo → OCR extracts all sales figures → manager provides only 2 manual inputs (payouts + actual cash) → system calculates variance and saves everything.

---

## Bot Flow (Revised)

```
REGISTERED
  → [photo received]
  → OCR via Claude Vision (extracts: register, date, cash_sales, card_sales, variance_from_register)
  → AWAITING_PAYOUTS
      bot: "💰 Retiros del cajón? Ingresa el monto o 0"
  → AWAITING_CASH
      bot: "💵 Efectivo real en el cajón?"
  → AWAITING_CONFIRMATION
      bot: shows full summary with calculated variance
  → [SI] → upload photo → save audit entry → save photo record → REGISTERED
  → [NO] → cancel → REGISTERED
```

### Variance formula
```
variance = actual_cash - (cash_from_z_report - payouts)
```

### State machine additions
New states added to `bot_state[user_id]`:
- `AWAITING_PAYOUTS` — stores `pending_data` (OCR result) + `pending_image_bytes`
- `AWAITING_CASH` — adds `payouts` to pending_data
- `AWAITING_CONFIRMATION` — adds `actual_cash`, `variance` (existing state, now with richer data)

## Data Model

### New table: `z_report_photos`

```sql
CREATE TABLE z_report_photos (
  id            BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  entry_id      BIGINT NOT NULL REFERENCES audits(id) ON DELETE CASCADE,
  store         TEXT NOT NULL,
  business_date DATE NOT NULL,
  register_id   TEXT NOT NULL,        -- e.g. "Reg 2"
  uploaded_by   TEXT NOT NULL,        -- telegram username
  uploaded_at   TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  storage_path  TEXT NOT NULL,        -- path in z-reports bucket
  content_type  TEXT NOT NULL DEFAULT 'image/jpeg',
  source        TEXT NOT NULL DEFAULT 'telegram'
);
CREATE INDEX ON z_report_photos(entry_id);
CREATE INDEX ON z_report_photos(store, business_date);
```

### Storage path format
```
z-reports/{store_slug}/{YYYY-MM-DD}/reg-{N}/{unix_timestamp}_{telegram_user_id}.jpg
```

### Backward compatibility
Existing entries with `z_report_image_path` in their `payload` JSONB are left unchanged. The old endpoint `/api/audit/<id>/zreport_image` continues to serve them. New entries use `z_report_photos` exclusively.

## API Changes

### New endpoints

| Method | Path | Auth | Purpose |
|--------|------|------|---------|
| GET | `/api/zreport/photos?entry_id=N` | session | Photo metadata for an entry (no URLs) |
| GET | `/api/zreport/signed_url?photo_id=N` | session | 1-hr signed URL — store-scoped IDOR check |

### Updated endpoint
`GET /api/list` — adds `photo_count: int` to each row (from z_report_photos).

### Security (IDOR prevention)
`/api/zreport/signed_url?photo_id=N`:
1. Fetch photo record, get its `store`
2. Check requesting user's session `store` or `role`
3. If user is not admin/super_admin and photo.store ≠ session.store → 403
4. Only then generate signed URL (1-hour expiry, server-side only)

## Frontend Changes

### History table
- Replace `d.z_report_image_path ? '📷' : ''`
  with `d.photo_count > 0 ? '📷(' + d.photo_count + ')' : ''`

### Modal
Existing `zreportModal` updated to show below the image:
- Store, Register, Date, Uploaded by, Upload time
- All metadata rendered via `element.textContent` — never `innerHTML` (XSS prevention)

### Error states
- "No hay imagen para esta entrada." — 404 from signed_url
- "No autorizado." — 403 from signed_url
- Network error fallback

## Security Model

### Assets
- Z-report photos (contain financial data)
- Signed URLs (time-limited access tokens)
- Audit entries (financial records)

### Trust boundaries
- Telegram → Railway server (bot webhook, server-side only)
- Browser → Railway server (authenticated session, HTTPS)
- Railway server → Supabase (service key, never exposed to browser)

### Threats & mitigations

| Threat | Mitigation |
|--------|-----------|
| IDOR on photos | Store-scoped check before generating signed URL |
| Path traversal | Storage path constructed server-side from validated OCR data |
| Stored XSS via metadata | `textContent` not `innerHTML` for all user-provided data |
| Secrets exposure | Supabase service key never in JS; signed URLs generated server-side only |
| File type abuse | Telegram photo messages are always JPEG; MIME allowlist enforced |
| Silent upload failure | `StorageUploadError` raised + logged; never saves empty path |
